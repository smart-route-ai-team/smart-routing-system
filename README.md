# AI-Based Smart NoC Routing System

> **A contribution to the Network-on-Chip (NoC) research project**  
> Multi-level intelligent packet routing for NoC architectures, powered by a professor-generated dataset of 30,000 UART / I2C / SPI protocol samples and a full Protocol Intelligence engine.

---

## Project Overview

Modern System-on-Chip (SoC) designs face a fundamental challenge: as chip densities grow into the ULSI era (well beyond 100 million transistors), global on-chip communication becomes the primary performance bottleneck rather than computation itself. This is the core motivation behind **Network-on-Chip (NoC)** — replacing ad-hoc bus-based interconnects with a scalable, packet-switched communication fabric.

This project implements an **AI-driven smart routing system** for NoC architectures that operates across three levels of intelligence — from classical graph algorithms to reinforcement learning. Every routing decision is grounded in a real, professor-generated protocol dataset rather than synthetic random values.

The theoretical foundation for this work is drawn from the survey *"A Survey of Research and Practices of Network-on-Chip"* (Bjerregaard & Mahadevan, ACM Computing Surveys, 2006), which establishes the key abstractions: **system**, **network adapter**, **network**, and **link layers**, along with routing strategies including store-and-forward, wormhole, and virtual cut-through.

## Routing Levels

### Level 3 — Basic Intelligent Routing (`level3/basic_routing.py`)

Implements classical graph-theoretic routing with dataset-derived edge weights.

- **K-Shortest Paths** using Yen's algorithm — finds the top-k least-cost paths
- **Dijkstra** — single-source shortest path with dataset latency as weight
- **A\*** — heuristic-guided search using node position coordinates
- Edge costs are directly drawn from `latency_cycles` in the dataset, replacing the random weights used in a naive baseline
- Each candidate path is annotated with its protocol type (UART / I2C / SPI) and a fitness score

### Level 4 — Adaptive Routing (`level4/adaptive_routing.py`)

Extends Level 3 with real-time congestion awareness.

- **Congestion detection** based on dataset-calibrated thresholds: an edge is flagged congested if `latency > 75th percentile` or `frame_err > 0.5`
- **Effective cost formula:**

  ```
  Effective Cost = base_weight × load_factor
                 + noise_penalty
                 + frame_err_penalty
                 + protocol_mismatch_penalty
  ```

- **PathMonitor** tracks per-route metrics across packets: attempt count, success count, congestion events, and average latency
- When the primary path becomes congested, the router automatically falls back to the next-best candidate

### Level 5 — Autonomous Routing via Q-Learning (`level5/qlearning_router.py`)

A reinforcement learning agent that learns optimal routing policies from experience.

- **Q-table** dimensions: `num_nodes × num_nodes`, representing the value of taking each possible next-hop from the current node
- **Reward function:** shaped by real edge `noise` and `frame_err` values from the dataset — noisy, error-prone edges yield negative rewards
- **CongestionPredictor:** a lightweight online linear model that tracks load history per edge and predicts future congestion before routing a packet
- Training runs for a configurable number of episodes (default: 500). The Q-table converges and a reward convergence chart shows learning stabilization
- After training, the agent selects the minimum-cost path autonomously using the learned policy

## Protocol Intelligence Engine (`utils/dataset_loader.py`)

A key contribution of this system is its ability to reason about which on-chip protocol is best suited for each network edge, based on the actual dataset statistics.

### Protocol Selection Rules

| Condition | Recommended Protocol | Reason |
|---|---|---|
| Payload ≤ 8 bits AND noise < 0.02 | **I2C** | Highest throughput-per-bit; drop rate only 0.92% |
| Payload > 8 bits (any duplex) | **SPI** | Handles large frames; lowest drop rate at 0.69% |
| Noise ≥ 0.05 (any payload) | **I2C or SPI** | UART drop rate rises sharply (~75%) at high noise |
| Latency budget < 460 cycles | **UART** | Lowest median latency at 451 cycles |
| Default / balanced workload | **I2C** | Best overall throughput-per-bit with minimal drops |

### Protocol Fitness Score

Each edge is assigned a fitness score between 0 and 1 indicating how well its current protocol matches the optimal choice for its traffic characteristics. A mismatch penalty is added to the routing cost:

```
Mismatch Penalty = (1.0 - fitness_score) × 2.0
```

This naturally steers the routing algorithms away from suboptimally configured edges.

### Mismatch Convergence via Traffic Refresh

When a network is first generated, some edges carry suboptimal protocols (reflecting real-world misconfiguration):

```
Initial state    → ~10–20 mismatches, Avg Fitness ≈ 0.87
After refresh 1  → ~6–8 mismatches,  Avg Fitness ≈ 0.92
After refresh 2  → ~2–3 mismatches,  Avg Fitness ≈ 0.96
After refresh 3  → 0 mismatches,     Avg Fitness ≈ 0.97
```

## Packet Drop and Retransmit Simulation (ARQ)

The dataset does not include a retransmit column. This system simulates **Automatic Repeat reQuest (ARQ)** on top of the dataset's `frame_err` values:

- A packet hop is considered **dropped** when the edge's `frame_err > 0`
- A **retransmit** is triggered automatically, up to 3 attempts
- Retransmit success probability is derived from the protocol's empirical drop rate
- If all 3 attempts fail, the packet is marked **permanently lost**
- All outcomes (delivered, recovered, permanently failed) are tracked per edge and reported in the dashboard

## ML Traffic Model (`utils/traffic_model.py`)

A scikit-learn model trained on all 30,000 dataset rows:

- **Regression task:** predict the network load percentage for a given edge's characteristics
- **Classification task:** predict whether a given edge will become congested (load > 70%)
- **Features used:** `latency_cycles`, `noise`, `frame_err`, `payload_bits`, `protocol` (one-hot encoded)
- Reported performance: MAE ≈ 9.86 on load prediction; Classifier Accuracy ≈ 87.2%

## Connection to NoC Theory

This implementation maps directly to the layered NoC architecture described in the survey literature:

| Survey Concept | Implementation |
|---|---|
| Routing nodes | NetworkX graph nodes; each stores congestion state |
| Links | Graph edges annotated with dataset `latency_cycles`, `noise`, `frame_err` |
| Network adapter | `utils/dataset_loader.py` — decouples protocol logic from routing |
| Wormhole routing analogy | Level 3 paths are selected end-to-end before transmission begins |
| Adaptive routing | Level 4's congestion-aware fallback mirrors dimension-order vs adaptive comparisons in literature |
| Virtual channel / QoS | Protocol fitness scoring provides a soft QoS mechanism — mismatched edges are costlier |
| Traffic characterization | Three protocol classes (UART / I2C / SPI) map to latency-critical, streaming, and best-effort traffic categories |
| Performance metrics | Latency (cycles), throughput (bps), drop rate, and jitter are all tracked per routing session |

## Design Decisions: Baseline vs This System

| Aspect | Naive Baseline | This System |
|---|---|---|
| Edge weights | `random.randint(1, 10)` | `latency_cycles` from professor's dataset |
| Load percentage | `random.randint(0, 100)` | Derived from `throughput_bps` |
| Congestion threshold | `load > 75%` | `latency > P75` OR `frame_err > 0.5` |
| Effective routing cost | Base weight only | Base + noise + frame_err + mismatch penalty |
| ML training data | Synthetic Gaussian | 30,000 real protocol samples |
| Protocol awareness | None | Full UART / I2C / SPI per-edge labeling with fitness scoring |
| Protocol optimization | None | Mismatch detection with iterative refresh convergence |
| Retransmit tracking | None | Full ARQ simulation with per-edge recovery statistics |
